# PURPOSE

이 문서는 `dev-jsLee.github.io` 저장소의 **핵심 목적과 운영 방침**을 정의한다.
모든 구조 결정, 페이지 구성, 서브 프로젝트 편입 여부는 이 문서를 기준으로 판단한다.

---

## 1. 한 줄 정의

> **개발자 jsLee의 포트폴리오용 GitHub Pages 저장소.**
> 직접 개발하거나 참여한 웹 프로젝트들의 **화면(프론트엔드)을 한곳에서 모아 보여주는 허브** 역할을 한다.

---

## 2. 핵심 컨셉

### 2.1 허브-위성(Hub & Spoke) 구조

- **허브(Hub)**: 이 저장소의 루트. 모든 서브 프로젝트로 진입하는 **랜딩(목차) 페이지**를 제공한다.
- **위성(Spokes)**: 각 서브 프로젝트. **Git Submodule**로 이 저장소 하위 폴더에 편입한다.

### 2.2 왜 Submodule인가

- 각 프로젝트는 **고유의 원본 저장소**를 유지해야 한다. (별도 이력, 별도 협업 가능성)
- 동시에 포트폴리오에서는 **그 시점의 스냅샷**을 고정해 서빙해야 한다.
- 코드를 복사/포크하지 않고 **참조만으로 통합**하기 위해 Submodule을 채택한다.

### 2.3 두 저장소의 역할 분리 (가장 중요한 운영 원칙)

| 구분 | 루트 저장소 (`dev-jsLee.github.io`) | 서브모듈 저장소 (각 프로젝트) |
| --- | --- | --- |
| 변경 권한 | **있음** — 자유롭게 커밋/푸시 | **없음(원칙적으로)** — 이 저장소 안에서는 수정하지 않는다 |
| 다루는 내용 | 랜딩 페이지, 카드 링크, 정적 자산, 서브모듈 포인터 | 해당 프로젝트의 실제 화면 코드 |
| Git 사용 목적 | 랜딩/하이퍼링크 관리 + 전체 구성의 **스냅샷 보존** | 원본 프로젝트의 자체 이력 |
| 업데이트 방식 | 직접 편집 후 커밋 | **`git pull`(또는 `git submodule update --remote`)로만 최신 반영** |

이 원칙을 어기지 않는다:

- 서브모듈 디렉토리 안의 파일을 직접 수정하지 않는다.
- 만약 서브 프로젝트의 코드를 고쳐야 한다면, **그 원본 저장소에서 직접 작업한 뒤** 이 저장소에서는 `pull`로만 가져온다.
- 루트 저장소의 커밋 중 서브모듈 관련 커밋은 본질적으로 **"어느 시점의 서브모듈을 가리키는가"의 포인터 변경**이다. 그 자체가 포트폴리오의 시점별 스냅샷 역할을 한다.

### 2.3 편입 대상의 조건

서브 프로젝트로 편입될 수 있는 조건:

- **순수 HTML / CSS / JS**로 동작할 것. (빌드 단계 없이 정적 파일 서빙만으로 화면이 뜨는 형태)
- 서버, 빌드 파이프라인, 환경변수, 비공개 키에 의존하지 않을 것.
- GitHub Pages 정적 호스팅으로 그대로 서빙 가능한 형태일 것.
- **포트폴리오 공개 자료로 적합한 수준**의 라이선스/저작권 상태일 것.

> 빌드가 필요한 프레임워크(React/Vue/Svelte 등) 프로젝트는 **빌드 산출물(dist)** 만 추출해 별도의 정적 저장소를 만든 뒤, 그 정적 저장소를 서브모듈로 편입한다. 소스 저장소를 직접 서브모듈로 붙이지 않는다.

---

## 3. 디렉토리 구조 (목표 상태)

```
dev-jsLee.github.io/
├── index.html              # 메인 랜딩(목차) 페이지 — 핵심 진입점
├── css/                    # 랜딩 페이지 전용 스타일
├── js/                     # 랜딩 페이지 전용 스크립트
├── assets/                 # 썸네일, 아이콘, OG 이미지 등 (예정)
│
├── projects/               # 서브모듈이 모이는 디렉토리 (예정)
│   ├── <project-a>/        # git submodule
│   ├── <project-b>/        # git submodule
│   └── ...
│
├── PURPOSE.md              # (본 문서) 저장소의 목적과 운영 방침
├── README.md               # 외부 방문자용 안내 — 별도 작성
└── .gitmodules             # 서브모듈 매니페스트
```

> 현재 루트의 `html/`은 초기 데모(TAMS) 잔존물이다. 새로운 구조로 이행할 때 `projects/` 하위 서브모듈 체계로 흡수하거나 정리한다.

## 4. 메인 랜딩 페이지 요구사항

`index.html`은 **이 저장소의 얼굴**이다. 단순한 링크 모음을 넘어 다음을 만족해야 한다.

### 4.1 콘텐츠 요구사항

- **개발자 소개 영역(Hero)**: 이름, 한 줄 소개, 핵심 기술 스택, 외부 링크(GitHub, 이메일 등).
- **프로젝트 카드 그리드**: 각 서브 프로젝트마다 1개 카드.
  - 프로젝트 이름
  - 한두 문장 설명
  - 사용 기술 태그(예: HTML / CSS / Vanilla JS / 특정 라이브러리)
  - 본인 역할(기획·디자인·프론트·풀스택 등)
  - 썸네일(있으면)
  - **해당 서브모듈의 메인 페이지로 이동하는 링크** (예: `projects/<name>/index.html`)
  - 원본 저장소 링크(선택)
- **푸터**: 저작권/연락처/저장소 링크.

### 4.2 분류 방식

프로젝트는 다음 중 하나의 축으로 분류해 섹션을 나눈다 (한 가지만 선택):

- 역할별 (개인 / 팀 참여 / 사이드 / 학습용)
- 기술별 (Vanilla / Library 활용 / 프레임워크 빌드 결과물)
- 도메인별 (관리 시스템 / 랜딩 / 게임·인터랙션 / 도구)

> 초기에는 **역할별 분류**를 기본으로 한다. 항목이 늘어나면 필요 시 재분류한다.

### 4.3 UX 요구사항

- 모바일 우선 반응형. 카드 그리드는 1~3열로 가변.
- 외부 의존성 최소화. CDN 사용은 허용하되 **빌드 단계 없이** 동작해야 한다.
- 다크 모드는 **선택 사항(필수 아님)**.
- 카드 클릭 시 새 탭 여부는 통일된 정책으로 정한다(기본: 같은 탭).

## 5. 서브 프로젝트 편입 절차 (운영 규칙)

### 5.1 새로 편입할 때

1. **편입 조건 점검** (3절의 정적 서빙 가능성, 라이선스 등)
2. `projects/<kebab-case-name>/` 경로로 **git submodule 추가**
3. 서브 프로젝트의 진입점이 `index.html`이 아니라면, **진입 경로를 명시**해 랜딩 카드 링크에 반영
4. 메인 `index.html`에 해당 프로젝트 카드 추가
5. (선택) 썸네일 이미지를 `assets/`에 추가
6. 커밋 메시지에 편입한 서브모듈명을 명시

### 5.2 최신 코드로 업데이트할 때

서브모듈의 새 변경사항을 포트폴리오에 반영하는 **유일한 경로는 pull**이다.

```bash
# 단일 서브모듈만 최신화
cd projects/<name>
git pull origin <기본브랜치>
cd ../..
git add projects/<name>
git commit -m "update submodule: <name>"

# 또는 모든 서브모듈을 한 번에
git submodule update --remote --merge
git add .
git commit -m "update submodules"
```

> 이 저장소의 커밋은 **"이 시점의 포트폴리오는 각 서브모듈을 이 커밋으로 가리킨다"** 는 스냅샷 그 자체다. 따라서 서브모듈 업데이트 후 루트에 커밋을 남기는 단계를 빠뜨리지 않는다.

### 5.3 하지 말 것

- 서브모듈 디렉토리 안의 파일을 이 저장소에서 직접 편집하지 않는다.
- 서브모듈 안에서 임시 커밋을 만들지 않는다. (필요한 변경은 원본 저장소에서 처리)
- 서브모듈을 detached HEAD 상태로 두고 작업하지 않는다. (pull은 항상 브랜치 위에서)

### 5.4 제거할 때

- 서브모듈 제거 (`git submodule deinit`, `.gitmodules` 정리, 캐시 디렉토리 제거)
- 랜딩 카드에서 해당 항목 제거

## 6. 기술/배포 제약

- **호스팅**: GitHub Pages (`main` 브랜치 기준 자동 배포).
- **빌드 단계 없음**: 저장소에 푸시된 정적 파일이 그대로 서빙된다.
- **루트 경로**: `https://dev-jslee.github.io/`
- **서브 프로젝트 경로**: `https://dev-jslee.github.io/projects/<name>/`
- **서브모듈 체크아웃**: GitHub Pages 빌드 시 서브모듈이 함께 받아져야 한다. 워크플로(예: `actions/checkout`)에서 `submodules: recursive` 옵션이 켜져 있어야 하며, 비공개 서브모듈은 사용하지 않는다(공개 저장소만 편입).

---

## 7. 비범위(Out of Scope)

이 저장소는 다음을 **하지 않는다**:

- 백엔드 API 호스팅
- 인증/세션 관리
- 데이터베이스 연동
- 빌드 파이프라인(번들러, 트랜스파일러) 운영
- 비공개 자료/사내 자료의 노출

위 기능이 필요한 프로젝트는 **별도 저장소에서 운영**하고, 본 저장소에는 **공개 가능한 화면 시연용 정적 빌드본만** 서브모듈로 가져온다.

---

## 8. 성공 기준

- 방문자가 랜딩 페이지에서 **3초 안에** 이 저장소의 정체("개발자 jsLee의 작업 모음")를 파악할 수 있다.
- 각 프로젝트 카드에서 **클릭 한 번**으로 해당 프로젝트의 실제 화면을 볼 수 있다.
- 새 프로젝트를 추가할 때 **랜딩 카드 1개 추가 + 서브모듈 1개 등록**만으로 마무리된다.
- 모든 서브 프로젝트가 **빌드 없이** 정적으로 서빙된다.
